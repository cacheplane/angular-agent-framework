# Content Agent (Design)

> Sub-spec 3 of the marketing umbrella. A Dawn app (`marketing/agent/`) that turns a `blog-merge` trigger into post-ready social drafts — an X thread (with a branded card) + a verbatim Dev.to syndication — written as a validated `DraftBundle` to the Cowork inbox. The LLM writes only the X thread; everything else is deterministic. Built on Dawn (cacheplane/dawnai) and invoked via `dawn dev` + `@langchain/langgraph-sdk`.

> **BLOCKED:** relies on a Dawn enhancement (upward `.env` discovery + `--env-file` flag) so `dawn dev` run from `marketing/agent/` picks up the repo-root `.env` (model API keys). Do that Dawn change first, then implement this. Interim fallback if needed: `dotenv -e ../../.env -- dawn dev`.

## 1. Goal

Turn a trigger into post-ready drafts in Brian's voice, dropped into `marketing/cowork/inbox/` for human approval. The agent never posts. v1 handles the `blog-merge` trigger end-to-end (the streaming-chat seed post is the test input).

## 2. Context

- Parent: `docs/superpowers/specs/marketing/2026-05-17-marketing-meta-design.md`. The `draft()` / `Trigger` / `DraftBundle` types are already declared in the `@threadplane/marketing-agent` skeleton.
- Consumes: `@threadplane/marketing-assets` (`renderCard`), `@threadplane/marketing-channels` (`validateDraft`, `Draft`). Both are merged (X + Dev.to adapters + card templates live).
- Built on **Dawn** (`cacheplane/dawnai`, local `~/repos/dawn`): a meta-framework for LangGraph. Routes under `src/app/` export `agent({ model, systemPrompt })`; Dawn discovers route-local `tools/*.ts`, wires a `ToolNode` + `bindTools` into a generated graph, runs locally via `dawn dev`, deploys to LangSmith. `@dawn-ai/*` published on npm (0.1.8).
- Dawn built-ins used: the **workspace capability** (`readFile`/`writeFile`/`listDir`, sandboxed to a workspace root, overridable) and **filesystem middleware** (`@dawn-ai/workspace`: `localFilesystem`, `compose`, `FilesystemMiddleware`, mirrored on `withFilesystemLogging`).

## 3. Requirements (decided during brainstorm)

- **Trigger (v1):** `blog-merge` only. (`release`/`cadence`/`cowork-prompt` deferred.)
- **Draft fidelity:** post-ready final copy (approve-as-is in Cowork, light edits at most).
- **Channels:** X + Dev.to.
- **X output:** a 3-6 tweet thread in Brian's voice; branded `x-card` PNG on tweet 1; blog link in the last tweet; each tweet ≤ 280.
- **Dev.to output:** verbatim syndication — strip cacheplane MDX components to clean markdown, map frontmatter (title/tags) → fields, set `canonical_url` to the cacheplane blog. No fresh rewrite. (Per the Dev.to adapter spec.)
- **No auto-posting:** output stops at the inbox; Cowork (next sub-spec) is the human gate.
- **Only the X thread uses the LLM.** Dev.to assembly, card render, validation, file I/O are deterministic.

## 4. Architecture — Dawn app with agent-orchestrates-everything (Approach B)

The skeleton `@threadplane/marketing-agent` becomes a Dawn app:

```
marketing/agent/
├── package.json            # Dawn app; deps below
├── langgraph.json          # generated by Dawn
├── dawn config
├── workspace/              # Dawn sandbox (agent confined here)
│   ├── .gitignore          # ignore source/ staging/ outbox/
│   └── (source/, staging/, outbox/ created at run time)
├── src/
│   ├── index.ts            # keeps exporting Trigger/DraftBundle/draft for consumers
│   ├── fs/
│   │   └── with-draft-bundle-validation.ts   # FilesystemMiddleware (the validation gate)
│   └── app/
│       └── draft/
│           ├── index.ts            # export default agent({ model, systemPrompt })
│           ├── system-prompt.ts    # orchestration + voice/positioning, authored inline
│           ├── state.ts            # lean route state
│           ├── tools/
│           │   ├── build-devto-draft.ts   # rawMdx + slug → validated Dev.to Draft (deterministic)
│           │   └── render-x-card.ts       # {title,subtitle} → staged PNG path (deterministic)
│           └── run.test.ts
└── scripts/
    └── draft-cli.ts        # stage → run (langgraph-sdk) → harvest
```

**Approach B (agent orchestrates):** the agent calls tools to do the work; deterministic logic lives in tool bodies (unit-testable). The agent's only creative job is writing the X thread.

**The agent is confined to `workspace/`** (Dawn's default sandbox). It never sees `apps/website/` or `marketing/cowork/`. The **trusted CLI** bridges in (stage inputs) and out (harvest outputs).

## 5. Validation gate — filesystem middleware (not a tool override)

Dawn's permission layer is path-based and auto-allows in-workspace writes, so it can't validate content. Instead we use Dawn's **filesystem middleware** (`FilesystemMiddleware = (next) => next'`), mirroring `withFilesystemLogging`:

```ts
// src/fs/with-draft-bundle-validation.ts
import type { FilesystemMiddleware } from '@dawn-ai/workspace';
import { validateDraft } from '@threadplane/marketing-channels';

export const withDraftBundleValidation: FilesystemMiddleware = (next) => ({
  ...next,
  async writeFile(path, content, ctx) {
    if (isOutboxPath(path)) {
      const bundle = parseAndValidateBundle(content); // zod shape + validateDraft per draft; throws on failure
      content = resolveStagedMedia(bundle);            // staged PNG path → base64 inline
    }
    return next.writeFile(path, content, ctx);         // "super" — localFilesystem does the real write
  },
});
```

Wired once via the app's workspace backend config:

```ts
backends: { filesystem: compose(withDraftBundleValidation)(localFilesystem()) }
```

The built-in `writeFile` tool (unmodified) now writes through the validating backend. A bad bundle throws before `next.writeFile` — nothing invalid lands. Hard gate, below the tool, can't be bypassed, reuses Dawn's real write.

## 6. Tools (deterministic; the LLM calls them)

**`build-devto-draft.ts`** — `{ rawMdx: string, slug: string }` → `{ draft: Draft }`. Parses frontmatter (gray-matter), strips MDX-only components (Callout/Steps/Tabs/etc.) to markdown, maps title→`article.title`, tags→`article.tags` (sanitized `^[a-z0-9]+$`, capped 4), body→`text`, canonical→`article.canonicalUrl` (`${SITE_ORIGIN}/blog/<slug>`), description→`article.description`. Runs `validateDraft(draft, { adapterId: 'devto' })` — throws on failure. No fs access (MDX arrives as an arg).

**`render-x-card.ts`** — `{ title: string, subtitle?: string }` → `{ stagedPath: string, alt: string }`. Calls `renderCard({ template: 'x-card', title, subtitle })`, writes the PNG to `workspace/staging/<uuid>.png`, returns the workspace-relative path + alt (defaults to title). No repo access (renderCard is a node dep; write is inside the sandbox).

Reads use Dawn's built-in `readFile`/`listDir`. The X thread is authored by the LLM directly (not a tool).

## 7. Route state + system prompt

**`state.ts`** — lean: `{ trigger: { kind: 'blog-merge'; slug: string }, sourcePath: string }`. Drafts don't live in state; they ride through tool args + the validated write.

**`system-prompt.ts`** — authored constant (Dawn convention: exported string → `agent({ systemPrompt })`). Contains the voice + positioning rules **authored inline** (canonical prompt-shaped copy; cites `docs/gtm/voice.md` + `messaging.md` as source) and the exact orchestration sequence:
1. `readFile(sourcePath)`.
2. `build-devto-draft({ rawMdx, slug })` → keep the returned Draft unmodified.
3. Write the X thread (3-6 tweets, voice, tweet 1 stands alone ≤280, last tweet ends with the blog URL, each ≤280).
4. `render-x-card({ title, subtitle })` → staged PNG path for tweet 1.
5. Assemble the `DraftBundle` JSON (id `<date>-<slug>`, drafts `[xDraft, devtoDraft]`, `xDraft.media` references the staged path) and `writeFile('outbox/<id>.json', ...)`.
6. On a validation error from `writeFile`, fix the offending draft and write again.

Never invent product claims; never exceed 280/tweet; never call network tools beyond those provided.

## 8. Invocation — `marketing draft` CLI (stage → run → harvest)

`scripts/draft-cli.ts`:

1. Parse `--trigger=blog-merge --slug=<slug>` (v1 rejects other triggers).
2. **Stage (trusted):** glob `apps/website/content/blog/*-<slug>.mdx`, copy → `workspace/source/<slug>.mdx`. Missing file → exit 1.
3. **Reachability:** check the local `dawn dev` endpoint (default `http://localhost:2024`); if down, print `pnpm --filter @threadplane/marketing-agent dawn:dev` and exit 1. (No auto-spawn in v1.)
4. **Run:** `@langchain/langgraph-sdk` `Client` → create thread, start run on `draft` graph with input `{ trigger, sourcePath: 'source/<slug>.mdx' }`, stream to completion.
5. **Harvest (trusted):** read `workspace/outbox/<id>.json`, re-validate (belt-and-suspenders), **move** → `marketing/cowork/inbox/<id>.json`, clean up `workspace/source` + `workspace/staging`. If no outbox file / re-validate fails → exit non-zero, leave workspace for debugging.

**Scripts:** `marketing/agent/package.json`: `"dawn:dev": "dawn dev"`, `"dawn:verify": "dawn verify"`, `"draft": "tsx scripts/draft-cli.ts"`. Root: `"marketing:draft": "pnpm --filter @threadplane/marketing-agent draft"`. (Once the Dawn env enhancement lands, `dawn dev` picks up the repo-root `.env` via upward discovery — no `dotenv-cli` wrapper needed.)

## 9. Data flow

```
marketing:draft --slug=X
  CLI: glob blog mdx → workspace/source/X.mdx ; langgraph-sdk start run
  graph (sandboxed):
    readFile source → build-devto-draft → write X thread → render-x-card (staging/<id>.png)
    → assemble bundle → writeFile outbox/<id>.json
        → withDraftBundleValidation: validate shape + each Draft; staged PNG → base64; else throw
  CLI: read outbox/<id>.json → re-validate → move to marketing/cowork/inbox/<id>.json → cleanup
  (later) /marketing skill reads inbox → human approves → channel adapters post
```

## 11. Error handling

- Missing blog file → CLI exits 1 pre-run.
- Oversized tweet / bad Dev.to tag → middleware rejects write with `rule`+`field`; agent rewrites; if it can't converge in the step budget, run ends with no inbox file → CLI exits non-zero with the final message.
- `render-x-card` (satori/resvg) failure → tool throws → CLI fails (no partial bundle).
- `dawn dev` down → CLI prints start command, exits 1.
- Harvest mismatch (no outbox file / re-validate fails) → CLI exits non-zero, workspace preserved.

## 12. Testing

- **Tool/middleware unit tests (no LLM):** `build-devto-draft` (strip + map + validate + tag sanitize); `render-x-card` (returns staged path; PNG exists); `withDraftBundleValidation` (valid passes to `next`; invalid/oversized/missing-channel throws before `next`; staged PNG → base64 resolution).
- **CLI unit tests:** staging globs+copies the right file; harvest moves+re-validates+cleans up; `dawn dev` not-reachable path.
- **One gated graph integration test** (env-flagged, real key): run the seed post through `dawn dev`; assert a valid bundle in a temp inbox with an X thread (≥3 parts, all ≤280) + a Dev.to draft. Skipped in normal CI.
- **`dawn verify`** as the build-gate (route + tools typecheck; graph artifact well-formed).

## 13. Scope

**In:** blog-merge trigger; X thread + branded card + Dev.to verbatim syndication; sandboxed Dawn app with staged I/O; validating fs middleware; `marketing draft` CLI; tool/middleware/CLI tests; one gated graph integration test.

**Out (follow-ups):** `release`/`cadence`/`cowork-prompt` triggers; auto-spawn `dawn dev`; LinkedIn/Reddit drafts; LangSmith deployment; Dawn AGENTS.md memory; multi-variant drafts; the cowork-loop approval skill (next sub-spec).

## 14. Risks

| # | Risk | Mitigation |
|--:|------|------------|
| 1 | Agent loops on validation | Dawn step budget bounds it; CLI fails cleanly; short tweets converge fast |
| 2 | Dawn app fights Nx tooling | Run via package scripts, not Nx targets; isolate under marketing/agent/ |
| 3 | `@dawn-ai/*` API churn (0.1.x) | Pin exact versions; use documented core (agent descriptor, workspace middleware, langgraph-sdk) |
| 4 | Model voice drift | Voice authored into system-prompt.ts; verified by gated integration test + human cowork review |
| 5 | Staged PNG path across processes | Workspace-relative paths; middleware resolves within workspace root; CLI cleans staging after harvest |
| 6 | Blocked on Dawn env feature | Sequenced first (this spec is `status: blocked`); interim `dotenv-cli` fallback exists if needed |
